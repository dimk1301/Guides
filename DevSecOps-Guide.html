01

Team Rules & Security Culture — NIST SSDF

Great tools don't matter if we don't have the right habits. This pipeline is built around the NIST Secure Software Development Framework (SSDF), which focuses on four key areas: preparing our team, protecting our code, producing secure software, and responding to vulnerabilities.

Ground Rules

  • Security Training — Every engineer completes annual secure coding training (OWASP Top 10) before getting CI/CD pipeline access.
  • Lock Every Account — MFA is mandatory on every SCM (GitHub/GitLab) account. Stolen or reused passwords are still the #1 way real breaches start, and this one setting blocks most of them before any tool below ever fires.
  • Vet New Dependencies Before They Land — Adding a brand-new third-party library takes 30 seconds of sanity-checking first: check the repo's last commit date and open issues, glance at its GitHub "Security" tab for known advisories, and run a quick dep-scan against it — the same tool our build pipeline already uses, works the same way regardless of language. Catching a bad dependency here is far cheaper than removing it after five other packages depend on it.
  • Don't Skip the Gates — Bypassing automated checks (Checkov, Gitleaks, Kyverno) is forbidden unless you have a documented exception signed by a Security Lead.
  • Treat Security Rules Like Code — Changing a security rule (Kyverno policy, Falco alert) goes through the same PR process as application code: two reviewers, one signed commit.
  • Make It Safe to Report Bugs — SECURITY.md in every repo, security.txt on every live app, so researchers report privately.

This isn't just good habit-forming — it's about to become the law. Starting in 2026–2027, the EU's Cyber Resilience Act (CRA) will require most of what's below for anyone shipping software into the EU. Building these habits now means we're ready, not scrambling, when that day comes.

Security Champions

  • Reviews security-related PRs
  • Maintains threat models and ASVS checklists
  • Acts as the first escalation point for security questions

Exception-Management Template

Every documented exception must include:

  • A clear description
  • Mitigations in place
  • A time-bound expiry (e.g., 30 days)
  • An automatic ticket in SecObserve — our central vulnerability dashboard, covered fully in Chapter 5

Annual Tabletop Exercise

Once per year, run a simulated incident (leaked secret, CVE exploit) and:

  • Validate SecObserve workflows
  • Verify MFA, backup restore, and Falco alerting
  • Log lessons learned

The Game Plan: "Shift Left"

Our core strategy is simple: catch bugs early. Instead of waiting for a security audit right before launch, our tools check for issues while you're designing, coding, and building.

We use free, community-driven tools and invest our time in setting them up well, rather than paying for expensive commercial software. Cluster rules follow the OWASP Kubernetes Security Testing Guide; app rules follow the OWASP ASVS.

Pipeline Controls

SBOM Generation — Every CI build must generate a CycloneDX or SPDX SBOM artifact, stored alongside the image. That same artifact is what makes app-level EOL checking free: the framework versions are already listed in it, so checking Django, Spring, or Express against endoflife.date needs no new scan — just a lookup against data you already produce.

Optional, if you want more confidence in your SBOMs later: a free tool called sbomqs can score how complete and well-formed a generated SBOM actually is — sbomqs score sbom.json. Not something to set up on day one — but worth a look if you ever suspect an SBOM is missing components rather than just showing "zero vulnerabilities found."

Branch-Protection Rules — Codified via Terraform or similar:

  • Requires signed commits
  • Requires 2 approvals
  • Requires a match between CODEOWNERS
  • Blocks force-pushes and deletion of protected branches

Pinned GitHub Actions — every marketplace action referenced in a workflow is pinned to a full-length commit SHA, not a tag (uses: org/action@a1b2c3d...); CI fails on a floating tag. Same principle as the digest-pinning rule for container images in Chapter 3 — a tag can move to point at compromised code without warning, a commit SHA can't.

Pre-Commit Hooks — Mandatory, enforced via:

  • A shared .pre-commit-config.yaml in the repo
  • CI validating that hooks ran (pre-commit run --all-files)

Starting Safe: Secure Container Images

Standard container images pulled by default from public hubs are bloated with extra tools like curl or bash. Hackers love these tools. We use stripped-down images instead:

  • dhi.io (Free Tier) — our standard choice for most apps. No shells, no package managers, updates automatically every week.
  • gcr.io/distroless — for compiled languages like Go or Rust that need bare-minimum libraries.

Image Comparison Matrix

FeatureStandard Public ImagesOur Standard (dhi.io Free)Future (dhi.io Enterprise)
UpdatesOccasional / quarterlyWeekly automated cyclesDaily / on-demand
Vulnerability FootprintVulnerable between releasesNear-zero CVEsNear-zero + SLAs
Hacker Friendly?Highly (ships with sh, apt)Zero (no shell, no utilities)Zero + FIPS crypto

Proving Our Images Are Ours (Signing)

We don't just trust images; we verify them.

  1. Build — Cosign digitally signs every image we make.
  2. Deploy — Kyverno checks the signature. No signature? The code doesn't deploy.
  3. Keys — Signing keys live safely in HashiCorp Vault.

Additional Image Controls

  • Pinned Digests — all image references use a digest, not a tag (image: dhi.io/app@sha256:...); CI fails on tags.
  • Base-Image Update Automation — automated weekly PRs update base images, with a minimum release age of at least a few days before a new version is even proposed. A same-day auto-bump is how several 2025 supply-chain attacks (npm's Shai-Hulud among them) got pulled into production before anyone noticed the compromised release.
  • Cosign Key Rotation — keys rotate annually; old keys revoked in Kyverno policy.
  • EOL Awareness — a monthly job checks the base OS and language runtime against endoflife.date and flags anything within 90 days of end-of-life. This is a different question from a CVE scan: an EOL version can show zero known vulnerabilities simply because no one is looking for new ones anymore, and the fix is usually a planned upgrade, not a patch. The same check applies one layer down to the host OS (Chapter 5) and one layer up to app frameworks (Chapter 2) — same question, asked at every layer that has its own release calendar.

Developer Tools — catching bugs before commit

The cheapest time to fix a bug is while you're writing the code. We run two layers of protection:

  1. On your laptop — fast tools that warn you instantly. You can bypass them locally while testing, but they exist to save you a headache later.
  2. In CI/CD — the unskippable automated gate. A hardcoded password or critical flaw will block the merge.

Repo Rules: signed commits, a CODEOWNERS file, a shared .gitignore baseline, and a second reviewer on every PR.

What "Blocking" Actually Means

"Unskippable gate" doesn't mean one tool stops everything — three different checkpoints block three different things, each enforced by a different mechanism underneath:

CheckpointWhat Gets BlockedEnforced By
Pre-commit hooksThe git commit itself, on your laptopLocal git hook (.pre-commit-config.yaml) — a failing check makes git commit exit with an error before the commit is even created
CI checks (Checkov, Gitleaks, dep-scan)The merge, not the commit — the commit can still exist on a branchGitHub branch protection — configured to require these checks pass before the "Merge" button is clickable at all
KyvernoThe deployment — has nothing to do with git or PRsKubernetes' own admission webhook system — intercepts kubectl apply at the API server and rejects it outright if the manifest violates policy

In short: commit → merge → deploy is one continuous chain, but each arrow in that chain has its own gatekeeper. A change has to clear all three to reach production.

Developer Environment Controls

  • IDE Config-as-Code — share SonarLint/ESLint/Bandit/gosec rules as code in .vscode/ or .idea/.
  • Secret-Detection in CI is Mandatory — CI fails on any Gitleaks hit; remediation requires rotating the secret, adding it to Vault, and updating the code.
  • Python Baseline — bandit -r . runs as a pre-commit hook and again in CI via .pre-commit-config.yaml, same unskippable pattern as Checkov and Gitleaks.
  • Go Baseline — gosec ./... runs alongside go vet as a pre-commit hook and again in CI; findings block the merge the same way a Gitleaks hit does.
  • Shell Script Baseline — shellcheck runs on every .sh file touched in a commit, catching injection-prone patterns in the CI glue and entrypoint scripts the platform relies on.
05

Our Security Toolkit & Where Everything Lives

This chapter maps every tool from Chapters 2–4 to the zone it runs in, then unpacks the two hubs everything ultimately reports into — and, since SecObserve is the one developers touch daily, a closer look at how its CVE workflow actually runs day-to-day.

In Short

Every tool from Chapters 2–4 runs at a specific checkpoint on the way from your laptop to a live server. This chapter shows where each one sits, the two dashboards everything eventually reports to, and exactly what happens when a vulnerability is found — you don't need to memorize every tool, just trust that your code passes through these checkpoints automatically.

Where Each Tool Runs

Think of a zone as a checkpoint — a specific place in the journey from writing code to a running pod where a specific kind of check happens. Every scanner from Chapters 2–4 runs in one of these, catching a specific class of problem at the point where fixing it is still cheap. Here's that path, checkpoint by checkpoint:

Planning Zone

OWASP ASVS and Threat Dragon figure out what needs to be secured before we start typing — the checklist and threat model a PR gets judged against later.

Workstation Zone — Your Laptop

SonarLint, Bearer CLI, Bandit, gosec, ShellCheck, and Gitleaks keep your local environment safe, catching the same classes of bug CI would — just before you've pushed anything.

Pipeline Zone — CI/CD

Checkov scans Terraform/Helm, Gitleaks double-checks for passwords, dep-scan --deep checks libraries and the base image OS. Nothing here is optional — this is the unskippable gate from Chapter 4.

Staging Zone

OWASP ZAP attacks our staging app to find runtime holes — broken auth flows, misconfigured headers, injection points — that no static scanner upstream can see, because they only exist once the app is actually running.

Edge Zone — Ingress / Load Balancer

Before traffic reaches the cluster we enforce HTTPS redirect, HSTS, CSP, and TLS 1.3, and publish security.txt. Baseline config, not a scan — set once per app, closing the exact holes ZAP would otherwise keep flagging in Staging.

Everything above catches problems we already know about — a bad pattern, a leaked secret, a missing header. But once the workload is live, a different kind of problem can appear: some things were fine at build time but aren't anymore, and others only show up once there's real traffic hitting the app. Two more zones keep watching for that:

Live Kubernetes Zone

Kyverno enforces rules at the admission door — think of it as a bouncer checking ID at the entrance, the one moment before a workload is allowed to actually start running. Falco is our broad monitoring camera across user space. Tetragon hooks into the Linux kernel via eBPF to enforce zero-trust runtime boundaries and instantly terminate unauthorized processes — a trust boundary here just means "the line between code we've already checked and code we haven't." Cilium/Calico segments the network. A scheduled dep-scan job continuously re-scans running images for newly disclosed CVEs and feeds results back into SecObserve — see the CVE Lifecycle below.

Compliance

OpenSCAP + STIG Viewer keeps the underlying host operating systems up to code and produces audit-ready reports auditors already recognize, on a cadence independent of any single app's release schedule. The same pass checks that host OS version itself against endoflife.date — a node can be fully patched and still be running a release that's about to lose support entirely.

Nine zones, but only two places where any of it actually gets read. Static findings (a scan result, a CVE, a config drift) and live telemetry (a syscall, a rejected deployment, a kernel-enforced kill) are very different shapes of data. So instead of forcing them into one tool, we split them across two purpose-built hubs.

Management Hubs

In plain terms: think of SecObserve as your inbox for things found before shipping, and OpenObserve as the live camera feed for what's happening right now.

SecObserve — Static Vulnerability Hub

Our central portal for point-in-time security scans, container vulnerability databases, and build-time compliance scores. Every scanner from the Pipeline Zone reports here, so there's one queue to check rather than nine.

For Developers

This is where the tool earns its keep day to day: a built-in rule engine auto-filters out known false positives and low-risk findings, so your queue stays signal, not noise. It also normalizes results from any scanner into one consistent view, so the dashboard stays familiar even as our tool list changes.

OpenObserve — Live Streaming Telemetry Hub

A single-binary, Rust-based system that captures and aggregates high-velocity streaming security logs and time-series metrics into highly compressed storage — no heavy database backends required. It's also the reason we could retire our old Java-based ELK/Loki stack (Chapter 6): same visibility, a fraction of the memory footprint.

Both hubs feed forward into later chapters: SecObserve's findings become the audit trail in Chapter 6, and OpenObserve's retention settings are what make the disaster-recovery drills in Chapter 7 provable rather than just assumed.

The CVE Lifecycle — management by exception, not by meeting

30-second version: most CVEs clear themselves automatically. Only a genuinely new HIGH/CRITICAL finding with no existing rule ever reaches a human.

How it actually works:

  1. Trigger — A commit (e.g., an automated dependency bump) kicks off a build-time scan.
  2. Auto-check — SecObserve compares the result against every rule already on file. Match found → it clears silently. No alert, no ticket, nobody looks at it.
  3. Escalation (rare) — Only a new HIGH/CRITICAL finding with no matching rule breaks the pattern. It goes to the Security Champion for one call: fix it, suppress it with a reason, or turn that reason into a rule so it never escalates again.
  4. Rollout gate — Non-production deployments always proceed, regardless of open findings. Production deployments only proceed once the verdict is clean.
  5. Ongoing — A scheduled dep-scan job keeps re-checking live deployments against newly disclosed CVEs, feeding results back into the same rule engine — not a separate process.
One Exception to "No One Has to Look"

If a newly disclosed CVE is flagged as actively exploited in the wild — not just theoretically possible, but actually being used in real attacks — it skips the normal queue entirely and goes straight to the Security Champion, regardless of severity rating or whether a rule already exists. This is the one case with a real clock attached: once CRA fully applies, actively-exploited vulnerabilities carry mandatory reporting deadlines, so treating this as routine backlog isn't an option.

How to read it: follow the grey arrows left to right for the normal path — most CVEs never leave that line. The colored dashed lines are the exceptions, explained in the legend below the diagram.

Most artifacts flow straight through, auto-cleared by rules already on file. Only a new HIGH/CRITICAL finding with no matching rule ever reaches the Security Champion — and resolving it once means the rule engine catches every future occurrence automatically.

Security Gate — Non-Productive vs Productive

Findings never block a test or staging rollout — developers need room to iterate. Only a productive rollout enforces the rule, e.g. "no unresolved CVE with criticality HIGH or higher." That gives the team the time between the first scan and the productive release to fix or formally suppress a finding, without slowing down everyday testing.

What You Actually Do

See a new finding in SecObserve? Check whether a rule already covers it before doing anything — most do. If it's genuinely new and HIGH or above, message the Security Champion directly instead of filing a ticket; the resolution is almost always one of three things: bump the dependency, suppress it with a documented reason, or — if it's likely to show up again — turn that reason into a rule. Whichever you pick, the second-reviewer habit from Chapter 1 still applies: someone else merges that change.

One note on the CVE workflow above before moving on: automation bots might suggest fixes on your PRs, but only human engineers can approve them — the Security Champion's judgment call is never automated away, only the noise around it is.

Infrastructure Controls — platform-team reference

Three standing controls keep the hubs above trustworthy. You don't manage these day to day — just know they exist:

  • Network segmentation is documented, not just assumed — Ingress → DMZ → App tier → Data tier, with Cilium/Calico policy placement marked on paper, so a reviewer can check a proposed change against the real trust boundaries in minutes.
  • Every policy decision is logged — a rejected deployment or a killed process is queryable in OpenObserve, never just something that happened and vanished.
  • Policies are reviewed annually against current CIS/ASVS baselines, so the rules enforcing zones like Edge and Live Kubernetes don't quietly drift out of date.

Proving It: Audit & Evidence

It's not enough to say we're secure; we have to prove it to auditors and ourselves. Every tool we use generates evidence, feeding into two centralized hubs: SecObserve (static, point-in-time build findings) and OpenObserve (live, high-velocity stream analytics).

SecObserve's biggest win for developers is trust: its rule engine auto-clears known false positives and accepted low-risk findings, so what's left in the queue is worth acting on. Less time triaging noise, more confidence in what the audit trail actually flags.

Access Control

SecObserve is locked down to protect this static evidence. Most of the team can update a finding; only a couple of trusted leads can delete one, so no one can quietly make a finding disappear. Every change needs a short written reason, saved automatically next to the finding, and sensitive cases can require a second sign-off — just like code review.

The live telemetry flowing into OpenObserve gets similar rigor: compressed, append-only files on secure storage mean application teams can't alter or delete operational security records — an unalterable trail for forensic audits.

Evidence Management

Automated Evidence Collection — CI/CD attaches to each release artifact in SecObserve: SBOM, dep-scan report, Checkov report, Gitleaks report, ZAP baseline scan.

  • High/Medium alerts — 2 years
  • Low/Informational — 1 year

Read-Only Auditor Role — external auditors get dedicated read-only access to SecObserve dashboards and OpenObserve SQL consoles, zero modify/delete permissions.

Bouncing Back — Resilience

Disaster recovery is a core part of security. Even with every prevention tool running, things can still go wrong — a bad deploy, a hardware failure, a real attack. What matters is whether we can get back the data and systems we lost.

WhatToolHow OftenHow We Prove It Works
Cluster State (etcd)Native etcd snapshottingDaily, off-clusterQuarterly restore onto a separate test cluster
App VolumesVeleroDailySame quarterly restore test
Application Data (DBs)pg_basebackup etc.Continuous + daily full backupQuarterly point-in-time restore, data verified
SecObserve recordsDB export, outside clusterNightlyRestore tests logged back into SecObserve
Network ContainmentCilium/Calico Network PoliciesAlways runningTraffic logs reviewed after any real incident
The One Rule

Backups are stored somewhere even our own cluster administrators can't delete or change. Full admin access to the cluster should still not reach yesterday's backup — that's the difference between a bad day and a permanent loss. In practice: WORM (Write Once, Read Many) object-lock on backup storage, enforced by the storage system itself, not Kubernetes permissions — so it still protects us even if K8s access is fully compromised.

Resilience Controls

  • Backup Test Automation — quarterly restore tests, automated via scripts, logged in SecObserve with pass/fail status.
  • RPO/RTO Definitions — Critical: RPO < 15 min, RTO < 4 hrs. Non-critical: RPO < 24 hrs, RTO < 24 hrs.

AI Assistants — helping, not taking over

In plain terms: the AI can suggest a fix, draft a policy, or group related alerts together — but it never gets to click the final "approve," "merge," or "deploy" button. That's always a person.

We use AI agents to move faster, but humans are always in control of the big decisions. This section is the floor, not the ceiling — anchored to the OWASP Top 10 for Agentic Applications (ASI Top 10) and two guiding ideas: Least-Agency (only the autonomy a task actually needs) and Strong Observability (always know what it did, why, under which identity).

Step-by-Step Implementation Checklist

Note on ordering: these steps follow rollout dependency, not the chapter order above — e.g., Image Provenance is step 6 even though Chapter 3 covered container images, because it depends on Vault and admission control being in place first. Each phase title links back to its source chapter.

1
Foundations & Code Security Gatekeeping → Ch.1 & Ch.4

Goal: make sure a bad commit can't quietly slip into the codebase. — Quick win, mostly configuration

  • Org-Wide MFA — across all SCM accounts.
  • Branch Protection — signed commits, two reviews, CODEOWNERS validation.
  • Local Security Tooling — SonarLint, Bearer CLI, Bandit, gosec, ShellCheck, Gitleaks; shared .gitignore baseline.
  • Dependency Vetting Habit — run dep-scan ad hoc before adding any new third-party library, alongside a quick check of its repo health and GitHub Security tab.
  • Pre-Commit Enforcement — shared config, CI verifies hooks ran.
  • Vulnerability Management Engine — stand up SecObserve as the central static reporting hub.
2
CI/CD Guardrails & Software Supply Chain → Ch.2

Goal: catch bad dependencies and missing SBOMs before they reach a build artifact. — 1–2 weeks

  • Pipeline Manifest Scanning — Checkov + Gitleaks in unskippable CI loops.
  • Dependency Triage — OWASP dep-scan --deep in build pipelines.
  • SBOM — CycloneDX or SPDX generated on every production build.
  • Dynamic Application Testing — OWASP ZAP in staging on every deploy.
3
Core Platform Ingress & Cluster Admission → Ch.5

Goal: stop untrusted traffic and non-compliant deployments at the front door. — 1–2 weeks

  • Edge Hardening — HTTPS redirect, HSTS, CSP, TLS 1.3 at ingress.
  • Vulnerability Disclosure — SECURITY.md per repo, security.txt on live properties.
  • Admission Control — Kyverno as primary cluster bouncer.
  • Network Micro-Segmentation — Cilium/Calico policies replace the flat network.
4
Unified Observability Infrastructure → Ch.5

Goal: give every later phase somewhere to send its logs and alerts. — ~1 week, mostly deployment

  • Platform Metric Scraping — OpenObserve Collector DaemonSet across the cluster.
  • Cluster Logging Architecture — deploy the chosen Log Shipper to feed OpenObserve.
5
Runtime Shielding & Kernel Enforcement → Ch.5

Goal: detect and stop malicious behavior in a container that's already running. — 2–3 weeks, includes a 7-day observation period
