What is Spec-Driven Development?

Spec-Driven Development (SDD) is the structural successor to "vibe coding" — the term Andrej Karpathy coined in February 2025 for the ad-hoc, prompt-and-pray loop of throwing natural-language instructions at a model until something works. Vibe coding was even crowned Collins Dictionary's Word of the Year for 2025. By 2026 it had also become clear that it doesn't hold up once real users, real security requirements, and real teammates are in the loop.

SDD is a methodology in which a written specification is treated as the primary, executable artifact of a software project, and code is a regenerable output produced from that spec by humans, AI agents, or both. The specification file, not the eventual codebase, becomes the single source of truth. Instead of describing a system brick-by-brick through an endless chat, you hand an autonomous agent a complete engineering blueprint. It builds against that blueprint, and when requirements change, you edit the spec and regenerate the code rather than hand-patch the output.

As Birgitta Böckeler of Thoughtworks puts it, SDD fundamentally means that "the specification is the single source of truth, and the code is a generated artifact." This shifts the developer's role from writing every line of code to maintaining the authoritative document that defines what the system should do.

The problem it actually solves

Improvising via chat is great for a prototype and dangerous for anything that has to survive contact with production. Models are extraordinary pattern-completers and unreliable mind-readers. Left unconstrained, ambiguity in your intent doesn't stay small — it compounds into architectures nobody chose on purpose, missing security controls, unhandled edge cases, and thin-to-nonexistent test coverage. Teams have taken to calling this failure mode "automating the creation of legacy mess." A written spec turns fuzzy intent into a deterministic contract the agent can't quietly drift away from. The lever for output quality shifts from prompt cleverness to spec quality.

CONTEXT

This distinction is echoed by the tooling itself: GitHub Spec Kit (see Sheet 13) frames it as specifications becoming living, executable artifacts rather than static documents — a traditional spec is read by humans, an SDD spec is meant to be consumed and acted on by an agent.

The Continuum of Commitment

Rolling SDD into a real engineering team isn't binary — it sits on a spectrum of how much authority you're willing to hand the spec. Most teams start at level one and drift toward level two; very few production shops run level three today. As Thoughtworks' Birgitta Böckeler notes, "all SDD approaches and definitions I've found are spec-first, but not all strive to be spec-anchored or spec-as-source."

LEVEL 1

Spec-First

A well-thought-out spec is fully drafted before engineering starts. The agent writes the baseline; from there, the human developer reviews, edits, and maintains the code by hand. Highest risk of the spec going stale the moment the first hotfix lands.

LEVEL 2

Spec-Anchored

Specs are treated as living, long-term artifacts that evolve alongside new features. Humans actively keep code and spec synchronized, so both remain aligned through the entire maintenance lifecycle — not just at launch.

LEVEL 3

Spec-As-Source

The specification is the only file humans edit. Generated code carries markers like // GENERATED FROM SPEC — DO NOT EDIT. Application code is treated as a compiled, transient byproduct — closer to how a build treats a .class file than how a team treats a repo.

WATCH FOR

Level 3 is the one vendors talk about most and teams actually run least. Treat it as a destination worth understanding, not a default starting point — most production teams get the bulk of SDD's value by simply reaching Level 2 and staying there.

DOES VIBE CODING DISAPPEAR?

No — it moves, it doesn't vanish. At Spec-First, ad-hoc prompting is completely normal after the initial build; the spec did its one job and steps aside. At Spec-Anchored, it's still normal for anything that doesn't change what the system does — you just triage first. At Spec-As-Source, the loose, exploratory energy shifts into the spec text itself: you never hand-edit generated code, but iterating on the spec is still just as conversational as vibe coding always was.

Core Workflow: The 4-Phase Loop

The SDD loop is a collaboration between you (the human) and the AI agent. Each phase has a clear division of labour: the agent drafts, you review and approve. Here's exactly who does what.

PhaseWhat the AI Agent doesWhat You (the Human) doesHandoff / Checkpoint
1. Specify Drafts the initial spec, asks clarifying questions, flags gaps Writes the initial description, reviews the drafted spec, answers questions, approves it You say "yes, proceed" — spec is locked
2. Plan Reads the spec, generates a technical architecture plan Reviews the plan, checks feasibility, approves or requests changes You say "yes, proceed" — plan is locked
3. Tasks Breaks the plan into an ordered checklist of micro-tasks Reviews the task list, checks completeness, approves it You say "yes, proceed" — task list is locked
4. Implement Executes the tasks one by one, writes code, runs tests Reviews the targeted diffs, validates against the spec, approves the merge You review each change; agent loops back if you request fixes
THE GOLDEN RULE

The agent never moves to the next phase without your explicit approval. You are the decision-maker. The agent is the executor. This is what prevents the "automated legacy mess" problem — you stay in control.

Greenfield vs. Brownfield: Where Phase 1 Actually Starts

The four phases are the same either way, but Phase 1 starts from a different place. Greenfield — there's no existing code to contradict, so the spec you write is genuinely the first source of truth. Brownfield — the codebase already has an implicit spec: however it actually behaves today, documented or not. Skip acknowledging that, and Phase 1 will happily draft requirements that quietly conflict with what's already shipped. Both tools have a specific fix for this — see Sheet 08 for the exact commands.

Phase 1 — Specify & Clarify

What the AI Agent does:

  • Reads your initial description (a few sentences about what you want to build).
  • Drafts a structured spec.md with functional requirements, constraints, and success criteria.
  • Runs a clarification pass — it identifies logical gaps, contradictions, and ambiguous statements.
  • Asks you specific questions to fill in the blanks ("What should happen if the API key is invalid?", "How many requests per second do you expect?").
  • Updates the spec based on your answers.

What You (the Human) do:

  • Write the initial description — just a few sentences about what you need.
  • Read the draft spec the agent produces.
  • Answer the agent's clarification questions.
  • Review the final spec and explicitly approve it before the agent moves to Phase 2.
EXAMPLE CONVERSATION

You: "Build a token-bucket API rate limiter middleware."
Agent: "I'll draft a spec. How many requests per second? What should the response be when rate-limited?"
You: "100 per second. Return 429 Too Many Requests."
Agent: "Here's the complete spec. Does this cover everything?"
You: "Yes, proceed." → Phase 1 complete.

Phase 2 — Plan

What the AI Agent does:

  • Parses the approved spec and generates a technical plan.md.
  • Decides on state management, dependency choices, data serialization formats, service-to-service communication, component and schema design.
  • Runs a risk assessment — flags security liabilities and integration pain points.
  • Produces a detailed architecture document that maps each requirement to a concrete implementation strategy.

What You (the Human) do:

  • Read the plan.md the agent generates.
  • Check if the technical approach makes sense for your context (e.g., does the chosen database fit your infrastructure?).
  • Flag any concerns or request changes.
  • Explicitly approve the plan before the agent moves to Phase 3.

Phase 3 — Tasks

What the AI Agent does:

  • Breaks the plan into an ordered checklist of micro‑tasks in tasks.md.
  • Each task is small and isolated enough to implement sequentially without collisions.
  • Each task includes its own definition of done and inline verification instructions.
  • Tasks map directly to the acceptance criteria from Phase 1.

What You (the Human) do:

  • Review the tasks.md checklist.
  • Check that all acceptance criteria are covered and the order makes sense.
  • Add or remove tasks if needed.
  • Explicitly approve the task list before the agent starts writing code.

Phase 4 — Implement

What the AI Agent does:

  • Executes the tasks one by one, in order.
  • For each task, it writes code — but not as a giant dump. It produces targeted diffs — small, focused changes.
  • Runs tests and linters against its own changes.
  • Continuously checks its work back against the Phase 1 spec. If it hits ambiguity, it re‑queries you.
  • If a task fails verification, it attempts to fix it or flags it for you.

What You (the Human) do:

  • Review each targeted diff — small changes are easier to understand and validate.
  • Test the code locally or in your staging environment.
  • If a change doesn't match the spec, reject it and the agent re‑works it.
  • Once all tasks are done and verified, approve the final merge.
THE VERIFIER PATTERN

The most underused refinement: assign a separate agent to verify the implementing agent's work rather than trusting it to self‑check. Treat "verifier" as a distinct role in your pipeline — the same way you wouldn't let one engineer be the only reviewer of their own pull request.

Two Kinds of Document: Feature Spec vs. Standing Context

Everything above — spec.md, plan.md, tasks.md — is temporary and localized. It exists for this one feature or bug fix, and once it ships you archive or freeze it. But every project also needs a second, different kind of document: standing rules that apply to every change, not just this one. As Birgitta Böckeler of Thoughtworks puts it: "general context are things like rules files, or high level descriptions of the product and the codebase… whereas specs [are] only relevant to the tasks that actually create or change that particular functionality." That standing-rules document is what the rest of this section covers — the Constitution.

The Constitution: Your Immutable Foundation

A Constitution defines non‑negotiable principles that apply to every change — testing philosophy, security baselines, code style, and opinionated technology choices. This ensures organizational standards are baked into every project from the start.

WHERE THIS TERM COMES FROM

"Constitution" isn't Kiro or Claude Code vocabulary — it originates with GitHub Spec Kit, another SDD tool, which uses it for the immutable rules file its workflow enforces. This guide borrows the term because it's become common shorthand across the SDD space. In Kiro it maps to steering files; in Claude Code it maps to CLAUDE.md — same concept, different file.

constitution.md (generic template)
# Project Constitution — immutable principles
- All database mutations must use explicit transaction boundaries.
- All public endpoints must include active rate-limiting middleware.
- Unit tests are required for any newly created utility method.
- Write tests before implementing any business logic (TDD discipline).

How the Constitution fits into the loop

  • You write it once at the start of the project.
  • The agent reads it at the beginning of every session — it's part of its permanent memory.
  • Every phase respects the Constitution: the agent won't propose a plan that violates it, and you use it as your own checklist during reviews.

How to implement the Constitution in each tool

  • Kiro CLI — Store your Constitution in steering files inside .kiro/steering/. For example, put architecture rules in tech.md, structural rules in structure.md, and product rules in product.md. Since steering files are always loaded in every interaction, they automatically enforce your immutable rules on every change — no extra configuration needed.
    .kiro/steering/tech.md — Kiro's Constitution
    # Technology Constitution
- All database mutations must use explicit transaction boundaries.
- All public endpoints must include rate-limiting middleware.
- Write tests before implementing any business logic.
  • Claude Code — Place your Constitution directly in CLAUDE.md, which is read at the start of every session. For larger rulebooks, keep a separate constitution.md and reference it in CLAUDE.md using the @constitution.md syntax, or simply include its contents. This ensures Claude applies your rules to every task it touches.
    CLAUDE.md — Claude's Constitution
    # Project Constitution (loaded every session)
- All database mutations must use explicit transaction boundaries.
- All public endpoints must include rate-limiting middleware.
- Write tests before implementing any business logic.

# Reference a separate constitution file if it grows large
@constitution.md
CONSTITUTION = PERSISTENT CONTEXT

Whether you use Kiro's steering files or Claude's CLAUDE.md, the Constitution is your team's living rulebook. It outlasts any single feature spec and ensures every AI session starts with the same non‑negotiable standards. Update it rarely — treat it like a real constitution, not a daily checklist.

HOW THIS PLAYS OUT IN PRACTICE

Kiro CLI: The built‑in /spec agent automates this entire loop. You run /spec new, the agent produces requirements.md. You review and type "proceed" — agent moves to design.md. You review and type "proceed" — agent moves to tasks.md. You review and type "proceed" — agent runs /spec run and executes the tasks, showing you each diff as it goes.

Claude Code: You bring your own workflow. Plugins like Specifico give you /specifico:spec, /specifico:plan, /specifico:tasks, /specifico:implement — each phase requires your explicit approval before moving to the next. You can also build custom slash commands with the same human‑first approval gates.

Writing Requirements with EARS Notation

The weakest link in any spec is still the sentence a human wrote. EARS (Easy Approach to Requirements Syntax) is the notation most SDD tools now lean on to keep those sentences testable — it originated in 2009 at Rolls-Royce, where engineers analyzing jet-engine control regulations noticed that good requirements all shared the same clause order. It has since been adopted by Airbus, Bosch, NASA, and Siemens, and since 2025 it's been baked directly into AI-native tools, most visibly Kiro's requirements phase.

EARS gives you five patterns. Each one forces you to be explicit about the thing vague requirements always hide: what triggers this, and under what conditions.

PatternKeywordTemplateExample
Ubiquitous— (none)The <system> shall <response>The system shall hash all passwords using bcrypt.
Event-DrivenWhenWhen <trigger>, the <system> shall <response>When a file upload exceeds 10MB, the system shall reject it with an error.
State-DrivenWhileWhile <state>, the <system> shall <response>While a password-reset link is valid, the system shall allow setting a new password.
Unwanted BehaviorIfIf <condition>, then the <system> shall <response>If an invalid card number is entered, then the system shall display a re-entry prompt.
Optional FeatureWhereWhere <feature present>, the <system> shall <response>Where two-factor auth is enabled, the system shall send a code via SMS.

These combine into complex requirements when a behavior genuinely depends on more than one condition:

requirements.md
# Complex requirement — state + event combined
While a user session is active, when the session token expires,
the system shall redirect the user to the login page and preserve their draft input.
WHY IT MATTERS FOR AI AGENTS

EARS' real payoff in an SDD pipeline isn't grammar — it's that each requirement becomes directly testable, with an unambiguous trigger and expected response an agent can decompose straight into a task and a test. It also front-loads the exact questions your clarification step needs to ask.

EARS works best for one trigger → one response. Stack more than two While/When clauses in a sentence and it turns into a puzzle instead of a requirement — switch to a decision table at that point. And some things aren't trigger-and-response at all: if it's about how components relate or fit together — architecture, service ownership, system layout — draw a diagram; if it's a standing rule with no trigger at all, one that's just always true, that's Constitution material (see Sheet 03), not a spec requirement.

Where the Spec Lives in Scrum & Kanban

The Big Misconception: "Waterfall in Markdown"

The loudest criticism of SDD is that it's just "waterfall in Markdown." That criticism is fair if you write a spec and then refuse to change it. But SDD becomes fully Agile when the spec is living — meaning it evolves as you learn.

The real test of agility is not whether you have a spec, it's whether the spec is allowed to change after development starts.

  • ❌ Frozen Spec – Locked down before coding, resistant to change. This is Big Design Up Front (BDUF) and is not Agile.
  • ✅ Living Spec – Updated during the sprint as new insights emerge. The AI regenerates code from the updated spec. This is Agile.
FOR MANAGERS

The Agile Manifesto prioritises "working software over comprehensive documentation" — it doesn't say documentation is useless. A living spec is working documentation because it directly drives AI to produce working software. It supports Agile's core value of "responding to change" better than a fixed plan ever could.

The Three Maturity Levels — and the One Antipattern

From Sheet 02, remember the three rungs. They map directly to how you use them in Agile:

LevelWhen to use itCost / Overhead
Spec-FirstMost day‑to‑day work inside a single sprint.Low — good for most features.
Spec-AnchoredWhen multiple teams or agents share the same contract (e.g., a public API).Medium — requires more coordination.
Spec-As-SourceExperiments, safety‑critical systems, or formal certification projects.High — full regeneration, heavy overhead.
THE ONE ANTIPATTERN TO AVOID

Paying the cost of a higher rung for a problem that sits at a lower one. If you turn a simple one‑line bug fix into a dozen acceptance criteria and multiple stories, you're over‑engineering the process. Keep the maturity level matched to the problem.

Where the Spec Belongs: In the Sprint, Not the Backlog

The most common integration mistake is writing the complete, finished spec in the Product Backlog. That's wrong.

  • In the Backlog: Keep only the intent — enough to understand the goal. Refinement is for shaping, not for writing full details.
  • In the Sprint: Develop the detailed spec (user stories, EARS requirements, acceptance criteria) as part of the implementation work.

As Yuval Yeret puts it: the team agrees on goals and context in the backlog, and develops the actual contract during the sprint, as part of doing the work. Scrum provides the container; it doesn't micromanage the spec cycle inside it.

In Scrum, the spec matures inside the Sprint, anchored by the Definition of Ready. In Kanban, the commitment point plays the same role.

The Definition of Ready (DoR): Agreement, Not Gate

In Scrum, the natural home for the spec is the Definition of Ready (DoR). The DoR is the team's shared understanding of when a story is ready to be pulled into a sprint. SDD's acceptance criteria naturally fill that need.

But: don't turn the DoR into a bureaucratic phase gate. Mike Cohn warns that the DoR can become "a dangerous step backward toward waterfall" if it blocks work rather than enabling collaboration. Keep it a quality agreement, not a weapon.

For Kanban: use the commitment point — the moment a task becomes a commitment to deliver. Before that, the task sits in Upstream Kanban for analysis and spec design, with WIP limits to avoid over‑producing specs that go stale.

Three Proven Integration Patterns

  • Just Enough Spec – Specify wide, not deep. Only enough to understand the shape of the solution.
  • Spec as Definition of Ready – Make the acceptance criteria the DoR itself.
  • Constitution + Spec – Keep a fixed rulebook (the Constitution) and a separate feature spec.

The Upside and the Downside

Teams that pair SDD with short feedback loops generally report real benefits — but mainly when the problem is complex and involves multiple people. For small changes, SDD can be overkill.

BUT THERE'S A DOWNSIDE

One practitioner reported a tenfold slowdown on a small CRUD feature under a rigid SDD workflow, because the tool generated more Markdown than code, all of which had to be reviewed.

Another warning: a top‑down mandate to adopt SDD reportedly backfired so badly that engineers signed a petition against it. The lesson: fit the level to the context, and introduce SDD as an experiment, not a decree.

Choosing the Right Level & Measuring Success

The selection rule in three sentences:

  • Use Spec-First for most everyday work.
  • Use Spec-Anchored with contract tests in CI when multiple teams share API contracts.
  • Use Spec-As-Source only experimentally or in safety‑critical systems.

Keep the backlog and refinement at the level of goal, develop the contract inside the sprint, and treat the DoR as a quality agreement, not a gate.

MEASURE — DON'T TRUST PERCEPTION

The biggest risk is psychological, not technical. In one study, developers expected a 24% speedup from AI, but actual measurement showed a 19% slowdown on a mature codebase. The gap between perception and reality is the real enemy.

Track these metrics:

  • Cycle Time – How long from start to finish.
  • Rework Rate – How often code needs to be changed due to misunderstandings.
  • Spec‑to‑Code Ratio – If your Markdown files grow faster than your code, you're over‑specifying.

If rework grows or specs bloat, reduce the batch size. And keep one owner for each spec to avoid drift.

THIS WEEK'S MOVE

Move spec‑writing from refinement into the sprint. Turn the DoR into a working agreement, not a checkpoint. Start measuring cycle time and rework rate — they'll tell you if AI is actually helping or just feeling like it.

SDD vs. TDD vs. BDD

SDD gets confused with test-driven and behavior-driven development constantly, because all three share the same instinct: define correctness before you write the implementation. The difference is what document is treated as the canonical, load-bearing artifact.

ApproachCanonical artifactCore questionWhere tests come from
TDDThe test file"Write the failing test first."Tests come first; implementation follows to satisfy them.
BDDThe Gherkin scenario"Describe behavior in business language first."Scenarios written collaboratively with stakeholders, then automated.
SDDThe full specification"Write intent, architecture, edge cases, and constraints — then let the agent produce code, tests, and docs."Generated from the spec, not written ahead of it by a human.

In practice, SDD subsumes a lot of what BDD was reaching for: EARS-style acceptance criteria are essentially Gherkin's stricter cousin, written so an LLM can parse them as cleanly as a person can. A healthy SDD workflow still produces real unit and integration tests — they just fall out of the spec rather than preceding it by convention.

Not everyone agrees SDD should go all the way to "code as byproduct," though. Some engineers hold a more conservative position: specs are a superior prompt, the way they were always meant to drive TDD, but the executable code — not the Markdown — remains the artifact you actually maintain long-term. That disagreement traces back to how far you're willing to travel down the maturity continuum in Sheet 02.

The Tooling Landscape: Kiro CLI & Claude Code

The ecosystem has moved fast from auto-complete helpers to full agentic SDD frameworks. As of mid-2026, the two tools actually worth your time for a production spec‑first workflow are Kiro CLI and Claude Code. They take different approaches to the same problem, and you can even use them together.

The hybrid moveStart a spec in Kiro CLI, continue it in the Kiro IDE, or hand it off to Claude Code for implementation. Both tools can read and write .kiro/specs/. Some teams report implementing Kiro‑authored specs with Claude Code is roughly 4× faster than using Kiro alone.

Kiro CLI — the opinionated spec agent

Kiro CLI's Spec agent is a built‑in agent that runs alongside your custom agents on a unified engine. You switch to it when you want the agent to think through requirements and design before writing code, then execute an implementation plan with verification between each task. The /spec commands let you list, create, resume, and run specs. When you run /spec new, the agent asks which type of spec you want — Build a Feature, Fix a Bug, or Quick Spec — and then produces the three files (requirements.md, design.md, tasks.md) with human review checkpoints between each phase. The specs are stored in .kiro/specs/, which is shared across all Kiro surfaces — start in the CLI, continue in the IDE.

Claude Code — bring your own SDD

Claude Code doesn't have a built‑in spec agent. Instead, you bring your own workflow through plugins, slash commands, hooks, and skills. The most complete plugin is Specifico, which enforces a strict spec → plan → tasks → execute → verify → merge loop. It produces structured artifacts (SPEC.md, PLAN.md, TASKS.md, STATE.json, DECISIONS.md) and maintains shared project memory in MEMORY.md and MEMORY.json for conflict detection across features.

You can also build your own workflow using Claude Code's .claude/commands/ directory and skills. A popular pattern is the Kiro‑style workflow implemented with custom slash commands like /kiro:spec-init, /kiro:spec-requirements, /kiro:spec-design, /kiro:spec-tasks. This enforces a 3‑phase approval process (Requirements → Design → Tasks → Implementation) where each phase requires human review and you cannot skip phases.

Claude Code extras: Skills & AGENTS.md

Beyond the spec workflow itself, Claude Code has two more building blocks worth knowing. AGENTS.md has become the de facto open standard for giving any coding agent — not just Claude Code — durable project context, so it's worth using instead of a Claude-only file if you work across multiple agent tools. Claude Code Skills are shareable workspace macros stored in .claude/skills/ that encode team conventions — file structure, test patterns, accessibility rules — as on-demand commands Claude can invoke directly, giving you a way to package recurring, spec-adjacent knowledge (like "how we structure API routes") without stuffing it into CLAUDE.md itself.

Starting on an existing codebase (brownfield)

Both tools have a built-in way to bootstrap the Constitution from a codebase that already exists, instead of writing it blank. Kiro: use Generate Steering Docs (a button in the IDE, and the same capability in the CLI and Web) to scan the existing repo and auto-draft product.md, tech.md, and structure.md from what's actually there. Claude Code: run /init in the repo before writing anything by hand — it detects your build system, test framework, and code patterns, and drafts a starter CLAUDE.md you refine from there. Either way, do this before your first spec, not after — it's what stops Phase 1 from drafting requirements that quietly contradict how the codebase already works.

PICKING ONE

If you want zero setup and an opinionated structure, start with Kiro CLI. If you already use Claude Code and want flexibility, start with Specifico or build custom slash commands. And if you want the best of both, start the spec in Kiro and implement it with Claude Code — the files are portable.

Vibe Coding vs. Spec-Driven: The Honest Call

SDD provides structural reliability, but it's genuine overhead for small tasks — turning a one-line fix into a web of user stories and acceptance criteria is its own anti-pattern, sometimes called "a sledgehammer to crack a nut."

Reach for Spec-Driven when:

  • You're writing production systems meant to scale and be maintained long-term.
  • Multiple engineers, or multiple parallel agents, work the same codebase at once.
  • You must enforce security profiles, data constraints, compliance rules, or design tokens precisely.
  • The system involves migrations, multi-service topologies, or intricate state machines.

Default to Vibe Coding when:

  • You're building throwaway code, isolated proofs-of-concept, or rapid wireframes.
  • The task is visual and needs fast, intuitive tweaking rather than a locked spec.
  • It's a minor bug fix — a full pipeline here is the sledgehammer problem above.
  • You're exploring an unfamiliar stack and don't yet know what you actually want.
A PRACTICAL TEST

One heuristic experienced SDD practitioners use cuts through the ambiguity fast: if you'd be annoyed to find the agent interpreted your requirement differently than you meant, write the spec. If a quick follow-up prompt would fix it just fine, skip the spec and prompt directly. It's a five-second gut check, and it's usually right.

Adoption Signals: Is This Actually Catching On?

SDD could easily be dismissed as another framework-of-the-month. Two things suggest otherwise: real production tooling has shipped around it (not just demos), and the industry-wide drop in developer trust toward unstructured AI-generated code is exactly the gap SDD's constraint layer is built to close.

Concrete case studies are starting to back the claims up: AWS has documented an internal notification feature that shipped in roughly two days of human effort as a Kiro-authored spec, versus an estimated two weeks the conventional way. And teams are combining tools: one report found that implementing Kiro-authored specs with Claude Code was roughly 4× faster than using Kiro alone.

On the tooling side, Kiro CLI 3.0 brings spec-driven development to the terminal with a built‑in Spec agent. The Claude Code ecosystem has grown rapidly, with plugins like Specifico and SDD providing structured, spec‑first workflows. The methodology has crossed from experimental to mainstream enough that DeepLearning.AI, in partnership with JetBrains, shipped a dedicated short course on spec-driven development with coding agents in late 2025.

Birgitta Böckeler's assessment captures the state of play: "Until I hear usage reports from people using them for a period of time on a 'real' codebase, I still have a lot of open questions about how this works in real life." The tools are evolving fast — they "might have already changed since I used them."

The Honest Limits

Every team adopting SDD should walk in aware of where it currently breaks down.

1. The brownfield adoption gap

SDD frameworks shine on greenfield projects where the agent builds from a clean slate. Dropping a full SDD pipeline onto a large legacy codebase in one go rarely works — it needs progressive, iterative adoption, feature by feature, not a big-bang rewrite. Both tools have a specific onboarding step for this (see Sheet 08), but even with it, treat the first few brownfield specs as trust-building exercises, not a reason to spec the entire legacy system at once.

2. The Markdown Monster risk

Adopt complex spec workflows without changing how your team actually collaborates, and you get documentation overhead instead of clarity. If a simple change spawns hundreds of verbose planning files, you're now spending more time reviewing prose than reviewing the code it produced. As Böckeler notes, "it is easy to end up with 100s of markdown files."

3. Semantic diffusion

As "spec-driven" gets popular, plenty of people use "spec" as a synonym for "a detailed prompt." That's not SDD — true SDD requires an execution engine that treats the specification as an actionable, verifiable source of truth, not just a longer prompt.

4. The heavy-upfront-planning critique

The Thoughtworks Technology Radar has placed SDD in its "Assess" ring — worth evaluating, not yet a default recommendation — and specifically flags a risk of the practice tilting back toward big up-front specification and big-bang releases, the same anti-pattern Agile spent two decades pushing teams away from. SDD is compatible with Agile when specs stay living, lightweight artifacts updated inside normal sprints — the moment they calcify into Waterfall-style documentation, you've lost the point.

5. The human-gates debate

There's a live disagreement about how much human review SDD pipelines should keep in the loop. One camp argues that as agents get faster and more capable, manual approval gates between phases become pure friction. The counter-argument, made forcefully by teams building for regulated environments: human gates were never there because agents can't do the work — they exist because production software carries risk, compliance, and legal accountability that someone has to own, and an agent's growing ability to create value scales in lockstep with its ability to amplify a mistake. Where you land on this depends heavily on what you're building and who's liable if it's wrong.

BOTTOM LINE

SDD is a genuine improvement over unstructured prompting for anything production-grade — but it's a discipline, not a silver bullet. It still requires someone to write a good spec, and a bad spec compiled perfectly is still a bad system.

Acronyms & Glossary

Navigating the SDD ecosystem means decoding a dense alphabet soup of terms. Here is a curated list of the most important acronyms and their meanings in the context of Spec-Driven Development.

AcronymFull FormContext / Meaning
SDDSpec-Driven DevelopmentThe core methodology where the specification is the single source of truth, and code is a generated artifact.
TDDTest-Driven DevelopmentA development process where you write a failing test first, then write just enough code to pass it.
BDDBehavior-Driven DevelopmentAn extension of TDD using shared language (Gherkin) to define behavior, often written as Given-When-Then.
EARSEasy Approach to Requirements SyntaxA structured notation (When/While/If/Where) for writing precise, testable requirements, heavily used in Kiro.
DoRDefinition of ReadyThe entry-quality criteria a story must meet before being pulled into a Scrum sprint — the natural anchor point for a maturing spec.
BDUFBig Design Up FrontA complete design agreed upon before implementation starts and defended by phase gates — the waterfall pattern SDD gets mistakenly conflated with when the spec is frozen rather than living.
WIP limitWork-in-Progress LimitA cap on the number of tasks allowed at a given Kanban stage, used in Upstream Kanban to prevent overproducing specs that go stale.
IDEIntegrated Development EnvironmentSoftware like VS Code, Cursor, or Kiro that provides comprehensive facilities to programmers.
CLICommand Line InterfaceA text-based interface used to run tools like Kiro CLI or Claude Code.
LLMLarge Language ModelThe AI model (e.g., Claude, GPT, Gemini) that interprets the spec and generates code.

Understanding these terms is essential for reading tool documentation and contributing to team discussions around SDD adoption.
