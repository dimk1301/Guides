What is Spec-Driven Development?

Spec-Driven Development (SDD) is the structural successor to "vibe coding" — the ad‑hoc, prompt‑and‑pray loop of throwing natural‑language instructions at a model until something works. SDD treats a written specification as the primary, executable artifact of a software project. Code is a regenerable output produced from that spec by AI agents. The specification file, not the eventual codebase, becomes the single source of truth.

As Birgitta Böckeler of Thoughtworks puts it, SDD fundamentally means that "the specification is the single source of truth, and the code is a generated artifact." This shifts the developer's role from writing every line of code to maintaining the authoritative document that defines what the system should do.

The problem it actually solves

Improvising via chat is great for a prototype and dangerous for anything that has to survive contact with production. Models are extraordinary pattern‑completers and unreliable mind‑readers. Left unconstrained, ambiguity in your intent compounds into architectures nobody chose on purpose, missing security controls, unhandled edge cases, and thin‑to‑nonexistent test coverage. A written spec turns fuzzy intent into a deterministic contract the agent can't quietly drift away from. The lever for output quality shifts from prompt cleverness to spec quality.[reference:0]

CONTEXT

Independent research is now catching up to the practice: a February 2026 arXiv paper, "Spec-Driven Development: From Code to Contract in the Age of AI," draws the line explicitly — a traditional spec is read by humans, an SDD spec executes as a validation gate that an agent's output has to pass.

Kiro CLI: The Built‑In Spec Agent

Kiro CLI's Spec agent is a built‑in agent that runs alongside your custom agents on a unified engine. You switch to it when you want the agent to think through requirements and design before writing code, then execute an implementation plan with verification between each task.[reference:12]

The /spec commands

CommandWhat it does
/specList existing specs in the workspace and select one[reference:13]
/spec new <name>Start a new spec — the agent switches to Spec mode and begins requirements[reference:14]
/spec <name>Resume an existing spec where you left off[reference:15]
/spec run <name>Execute all tasks in a spec's implementation plan autonomously[reference:16]

The three spec types

When you run /spec new, the agent asks what kind of spec you want[reference:17]:

  • Build a Feature — structured requirements, design, and implementation tasks
  • Fix a Bug — investigation, root cause analysis, and fix
  • Quick Spec — lightweight planning without full requirements formalization

How a Kiro spec works

The agent produces three files in .kiro/specs/<name>/[reference:18]:

  • requirements.md — structured acceptance criteria
  • design.md — technical design with architecture decisions and component breakdown
  • tasks.md — an ordered implementation plan with dependency tracking

You can review and edit these files between phases. The agent respects your changes.[reference:19]

Example Kiro session
# Start a new spec
/spec new auth-middleware
# Spec agent activated. Produces requirements.md

# Move to design
> Looks good, proceed to design.
# Agent produces design.md

# Execute the tasks
> Execute the tasks.
# Agent works through tasks.md sequentially, verifying between steps[reference:20]

Steering files: project‑wide context

Kiro uses steering files in .kiro/steering/ to provide persistent project knowledge[reference:21]. These are Markdown files that guide the AI with project‑wide rules and context[reference:22]:

  • product.md — product context and business objectives[reference:23]
  • tech.md — technology stack and architectural decisions[reference:24]
  • structure.md — file organization and code patterns[reference:25]

Steering files are always loaded in every interaction (or conditionally for specific file patterns). You can also reference them manually with @filename.md syntax.[reference:26]

PORTABLE SPECS

Specs are stored in .kiro/specs/ which is shared across all Kiro surfaces. Start a spec in the CLI, continue it in the IDE. The file format is identical.[reference:27]

Claude Code: Bring Your Own Spec Workflow

Claude Code doesn't have a built‑in spec agent like Kiro. Instead, you bring your own workflow through plugins, slash commands, hooks, and skills. This gives you more flexibility — and more responsibility to set up the structure yourself.[reference:28]

Option 1: Specifico — a complete SDD plugin

Specifico is a spec‑driven development system for Claude Code that enforces a structured, specification‑first workflow: spec → plan → tasks → execute → verify → merge.[reference:29]

Install Specifico
# Install Specifico in your project
npx specifico

# For an existing repo, run init first
/specifico:init

# Create a spec for a feature
/specifico:spec "build user authentication with JWT"

# Plan, break into tasks, and implement
/specifico:plan
/specifico:tasks
/specifico:implement

# Update project memory and merge
/specifico:memory-update
/specifico:merge

Specifico produces structured artifacts[reference:30]:

  • SPEC.md — human‑readable specification
  • SPEC.json — structured: problem, goals, entities, API endpoints, acceptance criteria
  • PLAN.md / PLAN.json — implementation plan
  • TASKS.md / TASKS.json — ordered task list with dependencies
  • STATE.json — current phase tracker
  • DECISIONS.md — append‑only decision log

It also maintains shared project memory in MEMORY.md (high‑level context) and MEMORY.json (machine‑checkable entity/API contracts) for conflict detection across features.[reference:31]

Option 2: Custom slash commands and skills

You can also build your own workflow using Claude Code's .claude/commands/ directory and skills in .claude/skills/.[reference:32] A popular pattern is the Kiro‑style workflow implemented with Claude Code slash commands, hooks, and agents[reference:33]:

Kiro‑style commands for Claude Code
# Phase 0: Steering (optional)
/kiro:steering               # Create/update steering documents

# Phase 1: Specification Creation
/kiro:spec-init              # Initialize spec with description
/kiro:spec-requirements      # Generate requirements document
/kiro:spec-design            # Interactive: "Have you reviewed requirements.md?"
/kiro:spec-tasks             # Interactive: confirms both requirements and design review

# Phase 2: Progress Tracking
/kiro:spec-status            # Check current progress and phases[reference:34]

This workflow enforces a 3‑phase approval process: Requirements → Design → Tasks → Implementation. Each phase requires human review (interactive prompt or manual), and you cannot skip phases — Design requires approved requirements; Tasks require approved design.[reference:35]

CLAUDE.md: your project's instruction file

CLAUDE.md is Claude Code's project‑level instruction file. It's read at the start of every session and shapes how Claude approaches your codebase.[reference:36] What goes in it[reference:37]:

  • Project context — what the project does, who uses it
  • Tech stack — frontend, backend, database, hosting
  • Build and test commands — Claude uses these to verify its own work
  • Coding standards — reference documents, don't duplicate
  • Project‑specific rules — what to do, what not to do

If you use multiple tools (Claude Code, Codex, Cursor), use AGENTS.md as the universal file and keep CLAUDE.md for Claude‑specific settings. Don't maintain both with overlapping content.[reference:38]

CLAUDE CODE SKILLS

Skills are shareable workspace macros in .claude/skills/ that encode team conventions — file structure, test patterns, accessibility rules — as on‑demand commands.[reference:39] There are plugins like SDD that provide 18 atomic, stack‑agnostic skills and a TDD implementation engine.[reference:40]

Writing Requirements with EARS Notation

The weakest link in any spec is still the sentence a human wrote. EARS (Easy Approach to Requirements Syntax) is the notation most SDD tools now lean on to keep those sentences testable — it originated in 2009 at Rolls‑Royce and has since been adopted by Airbus, Bosch, NASA, and Siemens. Kiro uses EARS natively in its requirements phase.

EARS gives you five patterns. Each one forces you to be explicit about the thing vague requirements always hide: what triggers this, and under what conditions.

PatternKeywordTemplateExample
Ubiquitous— (none)The <system> shall <response>The system shall hash all passwords using bcrypt.
Event‑DrivenWhenWhen <trigger>, the <system> shall <response>When a file upload exceeds 10MB, the system shall reject it with an error.
State‑DrivenWhileWhile <state>, the <system> shall <response>While a password‑reset link is valid, the system shall allow setting a new password.
Unwanted BehaviorIfIf <condition>, then the <system> shall <response>If an invalid card number is entered, then the system shall display a re‑entry prompt.
Optional FeatureWhereWhere <feature present>, the <system> shall <response>Where two‑factor auth is enabled, the system shall send a code via SMS.
WHY EARS WORKS FOR AI

EARS makes each requirement directly testable — an unambiguous trigger and expected response that an agent can decompose into a task and a test. It also front‑loads the exact questions your clarification step needs to ask: if you can't fill in the trigger or the state, that's the gap in your thinking, not a detail to skip.

Good vs. Bad: A Quick Example

  • ❌ Bad (vague): "The system should handle many requests quickly."
  • ✅ Good (EARS): When the system receives more than 1000 requests per second, it shall maintain a median response time under 200ms for 99% of requests.

EARS isn't universal, though — requirements with more than two or three preconditions get unreadable as a single sentence, and pure architectural constraints ("must use PostgreSQL," "must run on ARM") don't fit an event/response shape at all. Reach for a decision table or a diagram instead when you notice yourself stacking more than two While/When clauses.

Global Context vs. Feature Specs

Scaling SDD past a single feature means separating two kinds of documents that get conflated constantly: the spec for this change, and the standing rules that apply to every change.

  • Global Context (the "Company Handbook") — Persistent rules that apply to all projects and sessions. In Kiro, these are steering files (.kiro/steering/)[reference:41]. In Claude Code, these are CLAUDE.md or AGENTS.md[reference:42].
  • Feature Spec (the "Project Blueprint") — Temporary, localized to one task or bug fix. In Kiro, these live in .kiro/specs/<name>/[reference:43]. In Claude Code with Specifico, they live in specifico/<id>/[reference:44].

Kiro: Steering vs. Specs

  • Steering (.kiro/steering/) — Guide AI with project‑wide rules and context[reference:45]
  • Specs (.kiro/specs/) — Formalize development process for individual features[reference:46]

Claude Code: CLAUDE.md vs. feature specs

  • CLAUDE.md — Project‑level instructions read at the start of every session[reference:47]
  • Feature specs — Created per feature via plugins like Specifico or custom slash commands[reference:48]
BEST PRACTICE

Keep the instruction layer (steering / CLAUDE.md) and the feature layer (this week's spec) in separate files. Collapsing them is the single most common reason teams end up rewriting their "constitution" every sprint.

Where the Spec Lives in Scrum & Kanban

The most common objection to SDD is that it's just "waterfall in Markdown." That criticism is valid if your spec is frozen and never allowed to change. But SDD becomes fully Agile when the spec is living — meaning it evolves as you learn.

The real test of agility is not whether you have a spec, but whether that spec is allowed to change after development starts.

  • ❌ Frozen Spec – locked down before coding, resistant to change. This is Big Design Up Front (BDUF) and is not Agile.
  • ✅ Living Spec – updated during the sprint as new insights emerge. The AI then regenerates the code from the updated spec. This is Agile.
FOR MANAGERS

The Agile Manifesto says "working software over comprehensive documentation" — it prioritises working software, but it doesn't say documentation is useless. A living spec is a form of working documentation because it directly drives AI to produce working software. It supports Agile's core value of "responding to change" better than a fixed plan ever could.

Where the Spec Belongs: In the Sprint, Not the Backlog

Many teams make the error of writing the complete spec in the Product Backlog. That's wrong.

  • In the Backlog: keep only the intent — enough to understand the goal.
  • In the Sprint: develop the detailed spec (user stories, EARS requirements, acceptance criteria) as part of the implementation work.

As Yuval Yeret says: the team agrees on goals and context in the backlog, and develops the actual contract during the sprint, as part of doing the work. Scrum provides the container; it doesn't micromanage the spec cycle inside it, just as it doesn't micromanage individual TDD tests.

Definition of Ready: Agreement, Not Gate

In Scrum, the natural home for the spec is the Definition of Ready (DoR). The DoR is the team's shared understanding of when a story is ready to be pulled into a sprint. SDD's acceptance criteria naturally fill that need.

But: don't turn the DoR into a bureaucratic phase gate. Mike Cohn warns that the DoR can become "a dangerous step backward toward waterfall" if used to block work rather than to collaborate. Keep it a quality agreement, not a weapon.

Three Simple Integration Patterns

  • Just Enough Spec – specify wide, not deep. Only enough to understand the shape of the solution.
  • Spec as Definition of Ready – make the acceptance criteria the DoR itself.
  • Steering/CLAUDE.md + Spec – keep a fixed rulebook and a separate feature spec.

What the Data Says

+19% / -31% velocity gain and defect‑density drop in a quasi‑experiment pairing Scrum with an architectural spec. sMBSAP study, Colorado State, 2023
~-50% fewer defects on a semiconductor product after introducing a full reviewed spec. Single product‑line case report

MEASURE, DON'T TRUST PERCEPTION

The biggest risk is psychological, not technical. In one study, developers expected a 24% speedup from AI, but actual measurement showed a 19% slowdown on a mature codebase. The gap between perception and reality is the real enemy.

Track these metrics:

  • Cycle Time – how long from start to finish.
  • Rework Rate – how often code needs to be changed due to misunderstandings.
  • Spec‑to‑Code Ratio – if your Markdown files grow faster than your code, you're over‑specifying.
THIS WEEK'S MOVE

Move spec‑writing from refinement into the sprint. Turn the DoR into a working agreement, not a checkpoint. Start measuring cycle time and rework rate — they'll tell you if AI is actually helping or just feeling like it.

SDD vs. TDD vs. BDD

SDD gets confused with test‑driven and behavior‑driven development constantly, because all three share the same instinct: define correctness before you write the implementation. The difference is what document is treated as the canonical, load‑bearing artifact.

ApproachCanonical artifactCore questionWhere tests come from
TDDThe test file"Write the failing test first."Tests come first; implementation follows to satisfy them.
BDDThe Gherkin scenario"Describe behavior in business language first."Scenarios written collaboratively with stakeholders, then automated.
SDDThe full specification"Write intent, architecture, edge cases, and constraints — then let the agent produce code, tests, and docs."Generated from the spec, not written ahead of it by a human.

In practice, SDD subsumes a lot of what BDD was reaching for: EARS‑style acceptance criteria are essentially Gherkin's stricter cousin, written so an LLM can parse them as cleanly as a person can. A healthy SDD workflow still produces real unit and integration tests — they just fall out of the spec rather than preceding it by convention.

Not everyone agrees SDD should go all the way to "code as byproduct," though. Some engineers hold a more conservative position: specs are a superior prompt, the way they were always meant to drive TDD, but the executable code — not the Markdown — remains the artifact you actually maintain long‑term.

Vibe Coding vs. Spec‑Driven: The Honest Call

SDD provides structural reliability, but it's genuine overhead for small tasks — turning a one‑line fix into a web of user stories and acceptance criteria is its own anti‑pattern, sometimes called "a sledgehammer to crack a nut."

Reach for Spec‑Driven when:

  • You're writing production systems meant to scale and be maintained long‑term.
  • Multiple engineers, or multiple parallel agents, work the same codebase at once.
  • You must enforce security profiles, data constraints, compliance rules, or design tokens precisely.
  • The system involves migrations, multi‑service topologies, or intricate state machines.

Default to Vibe Coding when:

  • You're building throwaway code, isolated proofs‑of‑concept, or rapid wireframes.
  • The task is visual and needs fast, intuitive tweaking rather than a locked spec.
  • It's a minor bug fix — a full pipeline here is the sledgehammer problem above.
  • You're exploring an unfamiliar stack and don't yet know what you actually want.
A PRACTICAL TEST

One heuristic experienced SDD practitioners use cuts through the ambiguity fast: if you'd be annoyed to find the agent interpreted your requirement differently than you meant, write the spec. If a quick follow‑up prompt would fix it just fine, skip the spec and prompt directly. It's a five‑second gut check, and it's usually right.

Adoption Signals: Is This Actually Catching On?

SDD could easily be dismissed as another framework‑of‑the‑month. The data from the last year of AI‑coding adoption suggests otherwise — but it also shows the industry still has a trust problem SDD is explicitly trying to close.

9.8–42.1% of AI‑generated code contains vulnerabilities, depending on benchmark — the core problem SDD's constraint layer targets. Yan et al., 2025
90% / 13% of developers use AI at work, but only 13% use it across the entire SDLC — the gap SDD tooling is racing to close. JetBrains AI Pulse, Jan 2026
70%→60% drop in developer trust toward AI‑generated code accuracy from 2023–24 to 2025, even as usage kept climbing. Stack Overflow Developer Survey

Concrete case studies are starting to back the claims up: AWS has documented an internal notification feature that shipped in roughly two days of human effort as a Kiro‑authored spec, versus an estimated two weeks the conventional way. And teams are combining tools: one report found that implementing Kiro‑authored specs with Claude Code was roughly 4× faster than using Kiro alone.[reference:49]

On the tooling side, Kiro CLI 3.0 brings spec‑driven development to the terminal with a built‑in Spec agent.[reference:50] And the Claude Code ecosystem has grown rapidly, with plugins like Specifico and SDD providing structured, spec‑first workflows.[reference:51][reference:52]

Birgitta Böckeler's assessment captures the state of play: "Until I hear usage reports from people using them for a period of time on a 'real' codebase, I still have a lot of open questions about how this works in real life." The tools are evolving fast — they "might have already changed since I used them."

The Honest Limits

Every team adopting SDD should walk in aware of where it currently breaks down.

1. The brownfield adoption gap

SDD frameworks shine on greenfield projects where the agent builds from a clean slate. Dropping a full SDD pipeline onto a large legacy codebase in one go rarely works — it needs progressive, iterative adoption, feature by feature, not a big‑bang rewrite.

2. The Markdown Monster risk

Adopt complex spec workflows without changing how your team actually collaborates, and you get documentation overhead instead of clarity. If a simple change spawns hundreds of verbose planning files, you're now spending more time reviewing prose than reviewing the code it produced.

3. Semantic diffusion

As "spec‑driven" gets popular, plenty of people use "spec" as a synonym for "a detailed prompt." That's not SDD — true SDD requires an execution engine that treats the specification as an actionable, verifiable source of truth, not just a longer prompt.

4. The human‑gates debate

There's a live disagreement about how much human review SDD pipelines should keep in the loop. One camp argues that as agents get faster and more capable, manual approval gates between phases become pure friction. The counter‑argument, made forcefully by teams building for regulated environments: human gates were never there because agents can't do the work — they exist because production software carries risk, compliance, and legal accountability that someone has to own.

BOTTOM LINE

SDD is a genuine improvement over unstructured prompting for anything production‑grade — but it's a discipline, not a silver bullet. It still requires someone to write a good spec, and a bad spec compiled perfectly is still a bad system.
