Spec-Driven Development

Section 01

What is Spec-Driven Development?

The structural successor to "vibe coding" — the practice, named by Andrej Karpathy in February 2025, of throwing natural-language prompts at an AI agent until something works. Collins Dictionary crowned it 2025's Word of the Year. By early 2026, Karpathy himself was proposing a successor term, agentic engineering, for a more disciplined model where AI agents handle implementation while humans focus on architecture and verification. SDD is one concrete way of getting there.

Definition

Spec-Driven Development means documenting exactly what you need — functional requirements, non-functional constraints, technical boundaries, and success criteria — before a line of application code is written. The specification file, not the codebase, is the single source of truth. Instead of describing a system brick-by-brick through a chat window, you hand an autonomous coding agent a complete engineering blueprint. It builds against that spec, and when requirements change, you edit the spec and regenerate — you don't hand-patch source.

The problem it actually solves

Chat-driven prompting is excellent for isolated prototypes and dangerous for production systems, because generative models are master pattern-completers and hopeless mind-readers. Unclear intent doesn't stay small — it compounds into architecture nobody chose on purpose, missing security checks, silent edge cases, and no meaningful test coverage. Engineering teams have started calling this "automating the creation of legacy mess." A written spec turns fuzzy intent into a contract the agent can be held to — and it moves output quality from "how good was the prompt" to "how good was the spec."

Field note GitHub's own framing of Spec Kit puts it bluntly: coding agents are excellent pattern-matchers but literal-minded collaborators. Treating them like a search box gets you plausible-looking code that quietly solves the wrong problem. Treating them like a junior engineer with a signed-off spec gets you code you can actually trust.
Section 02

The continuum of commitment

SDD isn't one practice — it's a spectrum. Where your team lands determines how much of the "modify code by hand" habit you're willing to give up.

LEVEL 01

Spec-First

A spec is fully drafted before engineering starts; AI writes the baseline. Humans then edit the code directly. The spec is highly likely to go stale after the first release.

LEVEL 02

Spec-Anchored

Specs are living artifacts that evolve alongside features. Humans keep code and spec in sync deliberately, sprint over sprint, so both stay trustworthy.

LEVEL 03

Spec-As-Source

The spec is the only file a human edits. Generated code carries a header like // GENERATED FROM SPEC — DO NOT EDIT. Code is a compiled, disposable byproduct.

Where the industry actually is Almost every shipping SDD tool today operates at Level 1 or 2. Level 3 is closer to a thesis than a product: the honest, hands-on critique of the spec-as-source model is that generating code from a spec makes you dependent on a non-deterministic compiler — run the same spec twice and you can get two different implementations. It's a legitimate destination, not yet a safe default.
Section 03

The four-phase execution loop

Tool syntax differs, but nearly every production-grade SDD pipeline — GitHub Spec Kit, AWS Kiro, Tessl — collapses to the same rigorous loop.

Fig. 01 — SDD Execution Loop
1

Specify

Author a structured, behavior-oriented spec in Markdown. A /clarify pass has the agent flag logical gaps and contradictions before planning starts.

Written as functional requirements (often EARS notation: WHEN [event] THE SYSTEM SHALL [action], or Given–When–Then), technical constraints, and concrete success criteria.

2

Plan

The agent reads the finalized spec and produces a technical plan — architecture, state management, data schemas, dependency choices, and a risk pass over security and performance — without touching application code.

3

Tasks

The plan is broken into an ordered, dependency-isolated checklist. Each task gets an explicit definition of done. Some tools now run independent tasks concurrently in waves rather than one at a time.

4

Implement

The agent works the checklist item by item, producing targeted diffs you can actually review — instead of one unreviewable multi-file dump — validated continuously against the Phase 1 spec.

Section 04

Global context vs. feature specs

Scaling SDD past a single feature means separating what's true about this task from what's true about every task in this codebase. Feature specs are temporary — archived once the feature ships. Context files are permanent, and apply to every session.

Constitution

constitution.md — a global, non-negotiable rulebook (test-first, no inline styles, mandatory rate limiting) enforced on every session without re-typing it into every prompt. GitHub Spec Kit ships this as a first-class file, set up via a dedicated /speckit.constitution command.

Steering Files

Pioneered by AWS Kiro's .kiro/steering/ directory: modular, topic-scoped rule files (testing-standards.md, api-patterns.md) that get injected into context only when a matching file path is actually open — instead of one bloated global file.

Agent Hooks

Lifecycle-triggered background automations: save a spec, and a hook fires to generate tests, run linters, or write a compliant commit message — no human has to remember to ask.

Worth knowing

Tessl takes the context-sharing idea further with a public Spec Registry — over 10,000 pre-written "usage specs" for popular open-source libraries, installed the way you'd install a package, so an agent stops hallucinating APIs for a library it barely saw in training. It's a genuinely different problem from feature specs: not "what should this feature do," but "how does this dependency actually work."

Section 05

The tooling landscape, as it actually stands

Three tools define the space in mid-2026, each betting on a different maturity level from the ladder above.

Level 1–2 · Spec-First

GitHub Spec Kit

Open-source, MIT-licensed CLI (specify) that scaffolds .specify/ into any repo. Agent-neutral by design.

  • Seven slash commands: constitution → specify → clarify → plan → tasks → analyze → implement
  • 30+ agent integrations — Claude Code, Copilot, Cursor, Gemini CLI, Codex CLI, Windsurf, and more
  • Extensions and presets let teams bolt on compliance gates or swap methodology (Agile, Kanban, domain-driven design)
  • 111k+ GitHub stars; the most widely adopted SDD tool because it doesn't lock you to one vendor
v0.11.0 · June 2026 · github.com/github/spec-kit
Level 1–2 · Spec-First, IDE-native

AWS Kiro

An agentic IDE (built on Code OSS) where spec-writing is mandatory, not optional — the official successor to Amazon Q Developer, which exits support April 30, 2027.

  • Generates requirements.md, design.md, tasks.md using EARS notation before any code
  • Runs independent tasks concurrently in dependency-ordered "waves" instead of one at a time
  • Checks requirements for contradictions before planning, then validates behavior with property-based tests, not just example-based unit tests
  • Currently runs on Claude models via Amazon Bedrock; free tier plus paid Pro and Pro Max tiers
GA since May 7, 2026 · kiro.dev
Reaching for Level 3 · Spec-As-Source

Tessl Framework

The most aggressive bet in the category: specs, not code, become the artifact you maintain. Founded by Guy Podjarny (Snyk's former CEO).

  • Generated code is stamped // GENERATED FROM SPEC — DO NOT EDIT
  • Today's shipping product is spec-first, not spec-as-source: the agent asks clarifying questions, writes specs, pauses for human approval, then implements
  • Paired with the open Spec Registry (10,000+ library usage specs) to cut down on API hallucination separately from the methodology bet
  • The Framework itself remains in private beta as of mid-2026 — the spec-as-source vision is not yet generally available

Section 06

What the underlying model actually contributes

SDD tooling turns any capable LLM into a stricter compiler of intent — but the model still does the reasoning. A few things matter more than raw benchmark scores.

CONTEXT DEPTH

Holding the whole spec hierarchy

Long-context models can keep the constitution, steering files, and the active spec in view simultaneously instead of losing earlier constraints mid-session — this is where SDD workflows tend to fail quietly if the model's effective context is shorter than the team assumes.

STRUCTURED OUTPUT

Following EARS/GWT notation exactly

Reliable adherence to a fixed requirements grammar matters more than creativity here — a model that "helpfully" paraphrases the spec format breaks the automated clarify/analyze checks downstream.

TOOL & SKILL USE

Running the loop autonomously

Claude Code's .claude/skills/ folders, Kiro's steering files, and Spec Kit's skills-install mode all depend on the agent reliably invoking the right local context and re-checking it before every diff — not just at the start of the session.

Don't over-index on a specific version number Model line-ups move fast enough that any specific version named in a guide like this one is likely to be superseded within a quarter. Anthropic's current lineup (mid-2026) spans Claude Haiku, Sonnet, Opus, and a newer Mythos tier above Opus — check docs.claude.com for what's current before you pin a model in a constitution file. The same caution applies to OpenAI's o-series/GPT line and Google's Gemini line, both of which are also commonly wired into these pipelines as interchangeable orchestration engines.
Section 07

The developer's honest choice

SDD buys structural reliability at the cost of upfront overhead. Used on the wrong task, it's a sledgehammer for a one-line fix. Use this to decide per-task, not once per team.

Fig. 02 — Decision Path

Reach for spec-driven when

  • You're writing a system meant to scale and be maintained for years, not weeks.
  • Multiple engineers — or multiple autonomous agents — touch the same codebase concurrently.
  • Security profiles, data constraints, compliance rules, or design tokens must be enforced, not just requested.
  • The work involves migrations, multi-service topologies, or real backend state machines.

Default to vibe coding when

  • You're building throwaway code, a proof-of-concept, or a wireframe nobody will maintain.
  • The task is visual — fast, intuitive UI tweaks where iteration speed beats process.
  • It's a genuine one-line bug fix. A full spec/plan/tasks pipeline here is the "sledgehammer to crack a nut."
  • You're exploring a stack you don't know yet and need to learn by breaking things quickly.
Section 08

The honest limits, mid-2026

Every practitioner writing about this space in 2026 converges on the same handful of failure modes. Know them before you commit a team to the workflow.

RISK 01

The brownfield adoption gap

SDD shines on greenfield builds. Retrofitting it onto a large legacy codebase needs a progressive, incremental loop — teams that try a big-bang rewrite report spending hours correcting LLM-generated specs, with the process producing documentation volume rather than actual fidelity to the system.

RISK 02

The markdown monster

Generate a spec, plan, and task file for every trivial change and you'll spend more time reviewing repetitive Markdown than reviewing actual code. Spec-driven workflows also cost more per feature in raw token spend than vibe coding — often estimated at 20–40% more, offset by fewer wasted implementation cycles.

RISK 03

Specification drift

Tools that generate requirements.md and design.md once, upfront, don't automatically update them as implementation reveals new constraints — teams have reported the same "waterfall" drift SDD was supposed to fix, just one layer removed from the code.

RISK 04

Semantic diffusion

"Spec" is rapidly becoming a synonym for "detailed prompt." True SDD needs an execution engine that treats the specification as an actionable, checkable artifact — not just a longer chat message that happens to be saved to disk.

RISK 05

Rigid for exploratory work

Phased gates (specify → plan → tasks → implement) fit well-understood features. Research-heavy or genuinely iterative work fights the structure — you'll find yourself re-running /clarify and /plan more often than the workflow assumes.

RISK 06

Tooling churn

This category is moving fast enough that command surfaces change between minor releases — a real example: one popular toolkit removed its original CLI flag family outright in a mid-2026 release, breaking tutorials written months earlier. Pin tool versions the same way you'd pin a dependency.

Still compatible with Agile

None of the above means SDD is secretly Waterfall in a trench coat. Done well, specs are lightweight, living artifacts updated inside your normal sprint cadence — not a heavyweight requirements-document phase bolted on before "real" engineering starts.

Section 09

Quick start — GitHub Spec Kit

The fastest path to a working SDD pipeline today: free, MIT-licensed, and agent-neutral across 30+ coding assistants.

Step 1 — install & initialize
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git specify init my-sdd-project --integration claude
Step 2 — set your non-negotiables

Define your project constitution once — every session inherits it automatically.

# constitution.md - All database mutations must use explicit transaction boundaries. - All public endpoints require active rate-limiting middleware. - Unit tests are required for any newly created utility method.
Step 3 — run the compilation loop
# Phase 0: lock in project-wide rules (run once) /speckit.constitution # Phase 1: write the requirement /speckit.specify "Build a token-bucket API rate limiter middleware" # Phase 1b–3: clarify ambiguity, plan the architecture, break into tasks /speckit.clarify /speckit.plan /speckit.tasks /speckit.analyze // cross-checks spec, plan & tasks before you build # Phase 4: implement, task by task /speckit.implement
Before you run it Only install Spec Kit from github.com/github/spec-kit directly — the project explicitly warns that identically-named packages on PyPI are not maintained by the Spec Kit team.
