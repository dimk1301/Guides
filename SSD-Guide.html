What is Spec-Driven Development?

Spec-Driven Development (SDD) is the structural successor to "vibe coding" — the term Andrej Karpathy coined in February 2025 for the ad-hoc, prompt-and-pray loop of throwing natural-language instructions at a model until something works. Vibe coding was even crowned Collins Dictionary's Word of the Year for 2025. By 2026 it had also become clear that it doesn't hold up once real users, real security requirements, and real teammates are in the loop.

SDD means documenting exactly what you need — functional requirements, non-functional constraints, technical boundaries, and concrete success criteria — before a single line of application code exists. The specification file, not the eventual codebase, becomes the single source of truth. Instead of describing a system brick-by-brick through an endless chat, you hand an autonomous agent a complete engineering blueprint. It builds against that blueprint, and when requirements change, you edit the spec and regenerate the code rather than hand-patch the output.

As Birgitta Böckeler of Thoughtworks puts it, SDD fundamentally means that "the specification is the single source of truth, and the code is a generated artifact." This shifts the developer's role from writing every line of code to maintaining the authoritative document that defines what the system should do.

The problem it actually solves

Improvising via chat is great for a prototype and dangerous for anything that has to survive contact with production. Models are extraordinary pattern-completers and unreliable mind-readers. Left unconstrained, ambiguity in your intent doesn't stay small — it compounds into architectures nobody chose on purpose, missing security controls, unhandled edge cases, and thin-to-nonexistent test coverage. Teams have taken to calling this failure mode "automating the creation of legacy mess." A written spec turns fuzzy intent into a deterministic contract the agent can't quietly drift away from. The lever for output quality shifts from prompt cleverness to spec quality.

CONTEXT

Independent research is now catching up to the practice: a February 2026 arXiv paper, "Spec-Driven Development: From Code to Contract in the Age of AI," draws the line explicitly — a traditional spec is read by humans, an SDD spec executes as a validation gate that an agent's output has to pass.

The Continuum of Commitment

Rolling SDD into a real engineering team isn't binary — it sits on a spectrum of how much authority you're willing to hand the spec. Most teams start at level one and drift toward level two; very few production shops run level three today. As Thoughtworks' Birgitta Böckeler notes, "all SDD approaches and definitions I've found are spec-first, but not all strive to be spec-anchored or spec-as-source."

LEVEL 1

Spec-First

A well-thought-out spec is fully drafted before engineering starts. The agent writes the baseline; from there, the human developer reviews, edits, and maintains the code by hand. Highest risk of the spec going stale the moment the first hotfix lands.

LEVEL 2

Spec-Anchored

Specs are treated as living, long-term artifacts that evolve alongside new features. Humans actively keep code and spec synchronized, so both remain aligned through the entire maintenance lifecycle — not just at launch.

LEVEL 3

Spec-As-Source

The specification is the only file humans edit. Generated code carries markers like // GENERATED FROM SPEC — DO NOT EDIT. Application code is treated as a compiled, transient byproduct — closer to how a build treats a .class file than how a team treats a repo.

WATCH FOR

Level 3 is the one vendors talk about most and teams actually run least. Treat it as a destination worth understanding, not a default starting point — most production teams get the bulk of SDD's value by simply reaching Level 2 and staying there.

Core Workflow: The 4-Phase Loop

Tools differ in naming, but a production-grade SDD pipeline — the pattern popularized by GitHub Spec Kit and echoed by AWS Kiro, OpenSpec, and BMAD-METHOD — converges on the same rigorous execution loop.

The loop never truly ends — Implement validates continuously back against Specify, not just once at the end.

Phase 1 — Specify & Clarify

You author a structured, behavior-oriented Markdown artifact. Before planning starts, an automated clarification pass (/clarify) has the model parse the document for logical gaps, contradictions, and ambiguity — surfacing questions before they turn into wrong code. In Kiro, this phase typically produces a requirements.md file that lists user stories and their acceptance criteria.

  • Functional requirements — user stories and acceptance criteria, often written in EARS notation (more on that in Sheet 04) or GIVEN–WHEN–THEN form.
  • Technical constraints — data models, target schemas, third-party payloads, architectural layer boundaries.
  • Success criteria — concrete benchmarks, scale targets, security constraints, deterministic edge cases.

Phase 2 — Plan

The agent parses the finished spec and synthesizes a technical plan without touching a single line of application code: state management approach, dependency choices, data serialization formats, service-to-service communication, component and schema design, and an explicit risk pass over security liabilities and integration points.

Phase 3 — Tasks

The plan is broken into an ordered checklist of micro-engineering tasks — isolated enough to implement sequentially without collisions, each with its own definition of done and inline verification instructions.

Phase 4 — Implement

The agent runs the checklist item-by-item. You review targeted, focused diffs instead of a single unreadable multi-file dump, and every change is continuously checked back against the Phase 1 blueprint.

EMERGING PRACTICE

The most underused refinement to this loop: assigning a separate agent to verify the implementing agent's work rather than trusting it to self-check. Treat "verifier" as a distinct role in your pipeline, the same way you wouldn't let one engineer be the only reviewer of their own pull request.

The Constitution: Your Immutable Foundation

A constitution.md file defines non-negotiable principles that apply to every change — testing philosophy, security baselines, code style, and opinionated technology choices. This ensures organizational standards are baked into every project from the start. As GitHub's Spec Kit documentation puts it, the constitution is a "powerful rules file that is heavily used by the workflow" — it's the memory bank that persists across all AI coding sessions.

constitution.md
# Project Constitution — immutable principles
- All database mutations must use explicit transaction boundaries.
- All public endpoints must include active rate-limiting middleware.
- Unit tests are required for any newly created utility method.
- Write tests before implementing any business logic (TDD discipline).

Writing Requirements with EARS Notation

The weakest link in any spec is still the sentence a human wrote. EARS (Easy Approach to Requirements Syntax) is the notation most SDD tools now lean on to keep those sentences testable — it originated in 2009 at Rolls-Royce, where engineers analyzing jet-engine control regulations noticed that good requirements all shared the same clause order. It has since been adopted by Airbus, Bosch, NASA, and Siemens, and since 2025 it's been baked directly into AI-native tools, most visibly AWS Kiro's requirements phase.

EARS gives you five patterns. Each one forces you to be explicit about the thing vague requirements always hide: what triggers this, and under what conditions.

PatternKeywordTemplateExample
Ubiquitous— (none)The <system> shall <response>The system shall hash all passwords using bcrypt.
Event-DrivenWhenWhen <trigger>, the <system> shall <response>When a file upload exceeds 10MB, the system shall reject it with an error.
State-DrivenWhileWhile <state>, the <system> shall <response>While a password-reset link is valid, the system shall allow setting a new password.
Unwanted BehaviorIf / ThenIf <condition>, then the <system> shall <response>If an invalid card number is entered, then the system shall display a re-entry prompt.
Optional FeatureWhereWhere <feature present>, the <system> shall <response>Where two-factor auth is enabled, the system shall send a code via SMS.

These combine into complex requirements when a behavior genuinely depends on more than one condition:

requirements.md
# Complex requirement — state + event combined
While a user session is active, when the session token expires,
the system shall redirect the user to the login page and preserve their draft input.

In Kiro, requirements are structured as a list where each requirement represents a user story in "As a…" format with acceptance criteria in "GIVEN… WHEN… THEN…" format.

WHY IT MATTERS FOR AI AGENTS

EARS' real payoff in an SDD pipeline isn't grammar — it's that each requirement becomes directly testable, with an unambiguous trigger and expected response an agent can decompose straight into a task and a test. It also front-loads the exact questions your /clarify step needs to ask: if you can't fill in the trigger or the state, that's the gap in your thinking, not a detail to skip.

EARS isn't universal, though — requirements with more than two or three preconditions get unreadable as a single sentence, and pure architectural constraints ("must use PostgreSQL," "must run on ARM") don't fit an event/response shape at all. Reach for a decision table or a diagram instead when you notice yourself stacking more than two While/When clauses.

Global Context vs. Feature Specs

Scaling SDD past a single feature means separating two kinds of documents that get conflated constantly: the spec for this change, and the standing rules that apply to every change. As Birgitta Böckeler puts it: "general context are things like rules files, or high level descriptions of the product and the codebase. Some tools call this context a memory bank… whereas specs only relevant to the tasks that actually create or change that particular functionality."

  • Feature specifications — temporary, localized to one task or bug fix. Once it ships, archive or freeze it.
  • Memory banks / context files — persistent documents loaded into every session. They're your organization's standing source of truth, not this sprint's.

Pattern 1 — The Project Constitution

A global file acting as an uncompromising set of laws for every session, so nobody has to retype the same constraints into every prompt.

constitution.md
# Project Constitution
- All backend routes MUST enforce strict Zod runtime schema validation.
- Write Vitest unit tests before implementing any business logic.
- No inline styles — Tailwind utility classes only.

Pattern 2 — Context Steering Files

Pioneered by IDE platforms like AWS Kiro's .kiro/steering/ convention: instead of one massive global rule file, break rules into focused profiles (product.md, structure.md, tech.md) that only get injected into the agent's context when a matching file path is actually open.

Pattern 3 — Agent Hooks

Background agents triggered by workspace lifecycle events — save a spec, and a hook fires to write unit tests, run linters, or draft a compliant commit message, without you asking for it each time.

Pattern 4 — Claude Code Skills & AGENTS.md

AGENTS.md has become the de facto open standard for giving any coding agent durable project context. Claude Code's own CLAUDE.md plus scoped .claude/rules/ files follow the same idea — persistent instructions loaded automatically at the start of a session, kept deliberately separate from the feature spec itself. Additionally, Claude Code Skills are shareable workspace macros in .claude/skills/ that encode team conventions — file structure, test patterns, accessibility rules — as on-demand commands injected straight into the terminal runtime loop.

CLAUDE.md
# Project rules
- Read the feature spec before writing any code.
- Never expand scope beyond what the spec defines.
- Write tests before marking a task complete.
- If the implementation needs to drift from the spec, update the spec first.

Keep the instruction layer (constitution / CLAUDE.md) and the feature layer (this week's spec) in separate files. Collapsing them is the single most common reason teams end up rewriting their "constitution" every sprint.

SDD vs. TDD vs. BDD

SDD gets confused with test-driven and behavior-driven development constantly, because all three share the same instinct: define correctness before you write the implementation. The difference is what document is treated as the canonical, load-bearing artifact.

ApproachCanonical artifactCore questionWhere tests come from
TDDThe test file"Write the failing test first."Tests come first; implementation follows to satisfy them.
BDDThe Gherkin scenario"Describe behavior in business language first."Scenarios written collaboratively with stakeholders, then automated.
SDDThe full specification"Write intent, architecture, edge cases, and constraints — then let the agent produce code, tests, and docs."Generated from the spec, not written ahead of it by a human.

In practice, SDD subsumes a lot of what BDD was reaching for: EARS-style acceptance criteria are essentially Gherkin's stricter cousin, written so an LLM can parse them as cleanly as a person can. A healthy SDD workflow still produces real unit and integration tests — they just fall out of the spec rather than preceding it by convention.

Not everyone agrees SDD should go all the way to "code as byproduct," though. Some engineers hold a more conservative position: specs are a superior prompt, the way they were always meant to drive TDD, but the executable code — not the Markdown — remains the artifact you actually maintain long-term. That disagreement traces back to how far you're willing to travel down the maturity continuum in Sheet 02.

PICKING ONE

Three questions actually decide it: is your spec a living document or a static one-time plan? Are you coordinating one workspace or an entire org? And where does your team sit on the SDLC spectrum — a January 2026 JetBrains survey of 11,000 developers found 90% already use AI at work, but only 13% use it across the whole development lifecycle. If you're still early on that curve, start with Spec Kit or Cursor's rules before reaching for something as strict as Tessl.

As Birgitta Böckeler notes, evaluating SDD tools is time-consuming: "You would have to try them out with different sizes of problems, greenfield, brownfield, and really take the time to review and revise the intermediate artifacts with more than just a cursory glance."

LLM Support Matrix

SDD reframes an LLM from a creative text engine into a strict, deterministic compiler of human intent. How well that works still depends on the model underneath.

Anthropic Claude

Primary execution engine behind Claude Code, and frequently the model driving GitHub Spec Kit and AWS Kiro sessions. Large context windows let it hold an entire project's spec hierarchy at once, and it tends to handle spatial/UI layout constraints well without needing a visual feedback loop. Reusable macros live in .claude/skills/, forcing automated workspace checks and constitution validation before any diff gets proposed.

OpenAI (GPT / o-series)

Commonly wired in as an interchangeable orchestration engine inside Spec Kit and custom workspace extensions, leaning on structured JSON output and function-calling to check feature definitions against the central spec before code generation runs.

Google Gemini

Deployed natively in multi-file tools like Google Antigravity and inside multi-model platforms such as Cursor. Its long-context strength is most useful for ingesting a legacy codebase alongside a brand-new spec and mapping cross-module side effects before touching a file.

DON'T OVER-INDEX ON BENCHMARKS

Model capability is a fast-moving target — treat this matrix as a snapshot of architectural role, not a permanent leaderboard. Re-check current model support before betting a pipeline on any one vendor.

Vibe Coding vs. Spec-Driven: The Honest Call

SDD provides structural reliability, but it's genuine overhead for small tasks — turning a one-line fix into a web of user stories and acceptance criteria is its own anti-pattern, sometimes called "a sledgehammer to crack a nut."

Reach for Spec-Driven when:

  • You're writing production systems meant to scale and be maintained long-term.
  • Multiple engineers, or multiple parallel agents, work the same codebase at once.
  • You must enforce security profiles, data constraints, compliance rules, or design tokens precisely.
  • The system involves migrations, multi-service topologies, or intricate state machines.

Default to Vibe Coding when:

  • You're building throwaway code, isolated proofs-of-concept, or rapid wireframes.
  • The task is visual and needs fast, intuitive tweaking rather than a locked spec.
  • It's a minor bug fix — a full pipeline here is the sledgehammer problem above.
  • You're exploring an unfamiliar stack and don't yet know what you actually want.
A PRACTICAL TEST

One heuristic experienced SDD practitioners use cuts through the ambiguity fast: if you'd be annoyed to find the agent interpreted your requirement differently than you meant, write the spec. If a quick follow-up prompt would fix it just fine, skip the spec and prompt directly. It's a five-second gut check, and it's usually right.

Adoption Signals: Is This Actually Catching On?

SDD could easily be dismissed as another framework-of-the-month. The data from the last year of AI-coding adoption suggests otherwise — but it also shows the industry still has a trust problem SDD is explicitly trying to close.

9.8–42.1% of AI-generated code contains vulnerabilities, depending on benchmark — the core problem SDD's constraint layer targets. Yan et al., 2025
90% / 13% of developers use AI at work, but only 13% use it across the entire SDLC — the gap SDD tooling is racing to close. JetBrains AI Pulse, Jan 2026
70%→60% drop in developer trust toward AI-generated code accuracy from 2023–24 to 2025, even as usage kept climbing. Stack Overflow Developer Survey

Concrete case studies are starting to back the claims up: AWS has documented an internal notification feature that shipped in roughly two days of human effort as a Kiro-authored spec, versus an estimated two weeks the conventional way. GitHub reports that internal teams running Spec Kit see far fewer full "regenerate from scratch" cycles than ad-hoc prompting produces. And the methodology has crossed from experimental to mainstream enough that DeepLearning.AI, in partnership with JetBrains, shipped a dedicated short course on spec-driven development with coding agents in late 2025.

On the tooling side, GitHub Spec Kit alone had grown to roughly 88,000 stars and 129 releases by April 2026, with support for 28 named agent platforms — and OpenSpec, the newer proposal-first alternative, had reached about 52,000 stars by June 2026. That's not niche-framework growth; that's a workflow the ecosystem is actively converging on.

Birgitta Böckeler's assessment captures the state of play: "Until I hear usage reports from people using them for a period of time on a 'real' codebase, I still have a lot of open questions about how this works in real life." The tools are evolving fast — they "might have already changed since I used them."

The Honest Limits

Every team adopting SDD should walk in aware of where it currently breaks down.

1. The brownfield adoption gap

SDD frameworks shine on greenfield projects where the agent builds from a clean slate. Dropping a full SDD pipeline onto a large legacy codebase in one go rarely works — it needs progressive, iterative adoption, feature by feature, not a big-bang rewrite.

2. The Markdown Monster risk

Adopt complex spec workflows without changing how your team actually collaborates, and you get documentation overhead instead of clarity. If a simple change spawns hundreds of verbose planning files, you're now spending more time reviewing prose than reviewing the code it produced. As Böckeler notes, "it is easy to end up with 100s of markdown files."

3. Semantic diffusion

As "spec-driven" gets popular, plenty of people use "spec" as a synonym for "a detailed prompt." That's not SDD — true SDD requires an execution engine that treats the specification as an actionable, verifiable source of truth, not just a longer prompt.

4. The heavy-upfront-planning critique

The Thoughtworks Technology Radar has placed SDD in its "Assess" ring — worth evaluating, not yet a default recommendation — and specifically flags a risk of the practice tilting back toward big up-front specification and big-bang releases, the same anti-pattern Agile spent two decades pushing teams away from. SDD is compatible with Agile when specs stay living, lightweight artifacts updated inside normal sprints — the moment they calcify into Waterfall-style documentation, you've lost the point.

5. The human-gates debate

There's a live disagreement about how much human review SDD pipelines should keep in the loop. One camp argues that as agents get faster and more capable, manual approval gates between phases become pure friction. The counter-argument, made forcefully by teams building for regulated environments: human gates were never there because agents can't do the work — they exist because production software carries risk, compliance, and legal accountability that someone has to own, and an agent's growing ability to create value scales in lockstep with its ability to amplify a mistake. Where you land on this depends heavily on what you're building and who's liable if it's wrong.

BOTTOM LINE

SDD is a genuine improvement over unstructured prompting for anything production-grade — but it's a discipline, not a silver bullet. It still requires someone to write a good spec, and a bad spec compiled perfectly is still a bad system.

Quick Start (GitHub Spec Kit)

The fastest path to a working SDD pipeline today, using uvx:

Step 1 — Initialize the workspace

terminal
uvx --from git+https://github.com/github/spec-kit.git specify init my-sdd-project

Step 2 — Write your constitution

constitution.md
# Project Constitution
- All database mutations must use explicit transaction boundaries.
- All public endpoints must include active rate-limiting middleware.
- Unit tests are required for any newly created utility method.

Step 3 — Run the compilation loop

terminal
# Phase 1 — initialize the spec
/speckit.specify "Build a token-bucket API rate limiter middleware"

# Phase 2 & 3 — clarify, plan, break into tasks
/speckit.clarify
/speckit.plan
/speckit.tasks

# Phase 4 — implement sequentially against the spec
/speckit.implement
FIRST PROJECT TIP

Run your first pass on something small and real — a rate limiter, a webhook validator, a single CRUD endpoint. Learn the rhythm of /clarify catching your own ambiguity before you trust it on anything with production stakes.

Acronyms & Glossary

Navigating the SDD ecosystem means decoding a dense alphabet soup of terms. Here is a curated list of the most important acronyms and their meanings in the context of Spec-Driven Development.

AcronymFull FormContext / Meaning
SDDSpec-Driven DevelopmentThe core methodology where the specification is the single source of truth, and code is a generated artifact.
TDDTest-Driven DevelopmentA development process where you write a failing test first, then write just enough code to pass it.
BDDBehavior-Driven DevelopmentAn extension of TDD using shared language (Gherkin) to define behavior, often written as Given-When-Then.
EARSEasy Approach to Requirements SyntaxA structured notation (When/While/If/Where) for writing precise, testable requirements, heavily used in Kiro.
PRDProduct Requirements DocumentA business-facing document outlining what a product should do, often a precursor to an SDD spec.
IDEIntegrated Development EnvironmentSoftware like VS Code, Cursor, or Kiro that provides comprehensive facilities to programmers.
CLICommand Line InterfaceA text-based interface used to run tools like GitHub Spec Kit or OpenSpec.
MCPModel Context ProtocolA protocol used by tools like Tessl to provide structured context to Large Language Models.
LLMLarge Language ModelThe AI model (e.g., Claude, GPT, Gemini) that interprets the spec and generates code.
AIArtificial IntelligenceThe overarching field; in this context, specifically the agentic coding assistants.
UI/UXUser Interface / User ExperienceRefers to the visual design and usability of the software being built, often requiring specific constraints in specs.

Understanding these terms is essential for reading tool documentation and contributing to team discussions around SDD adoption.
